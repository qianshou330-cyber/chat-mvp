# v0.7.3-beta Search Pagination and Request Reliability

`v0.7.3-beta` adds paged server-search results and safer request handling on top of the v0.7 message history layer. It does not add channels, bots, voice/video calls, end-to-end encryption, or a dedicated enterprise search service.

## Changes

- Supabase migration: `20260620000000_message_search_pagination.sql`.
- New RPC: `search_messages_v3`.
- `search_messages_v3` keeps the v2 filter shape and adds cursor pagination with `before_created_at` and `before_message_id`.
- Global search and current-conversation search support “加载更多结果”.
- Search input uses 300 ms debounce.
- Each request has a client-side request id so stale responses do not overwrite newer query results.
- Search failures stay inside the current search panel with a retry action.

## Validation

- Run local checks: `npm run lint`, `npm run test -- --run`, `npm run build`, `npm run e2e`.
- In Supabase, run `20260620000000_message_search_pagination.sql` and confirm `search_messages_v3` exists.
- Production smoke should cover Chinese keyword search, type/sender/date filters, load more, result navigation, and A/B/C permission regression.
- Temporary smoke accounts should use `codex.v073.*@example.com`; test uploads should use `codex-v073-*`.

## Boundaries

- The RPC still uses Supabase contains matching, not Meilisearch, Typesense, Elasticsearch, or enterprise full-text search.
- Hidden attachments and deleted messages must not appear in any page of results.
- Search error logs must not store the raw search keyword.
