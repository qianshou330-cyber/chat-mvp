# 5 人先导试用启动清单

## 目标

以 `v0.6.8-beta` 为基线，用 5 个真实核心用户验证公司内部试用的核心路径。先导阶段只收集和修复 P0/P1，不引入新功能。`v0.6.8-beta` 必须先完成真实 A/B/C 生产 smoke，再作为 20-30 人试用启动基线。

## 启动前准备

- Production URL：`https://chat-mvp-tau.vercel.app`
- 基线版本：`v0.6.8-beta`
- 管理员账号：选择 1 个 owner 作为正式试用管理员。
- 群聊：创建或确认 1 个正式试用群聊，命名为公司或团队名称。
- 试用用户：准备 5 个真实邮箱，要求每个用户先完成注册和登录。
- 安全说明：把 `docs/company-trial-safety.md` 发给试用者，提醒不要发送敏感信息。
- 反馈入口：所有问题进入 GitHub Issues，按 P0/P1/P2/P3 标注。

## 管理员启动步骤

1. 使用 owner 账号登录生产站点。
2. 创建或打开正式试用群聊。
3. 进入群详情页，通过邮箱添加 5 个已注册用户；系统会自动把成员纳入当前工作区权限边界。
4. 至少设置 1 名群 admin，其余用户保持 member。
5. 发送一条欢迎消息，确认成员能看到群聊和消息。
6. owner/admin 在群详情页打开“更多管理”，确认“群管理记录”能看到成员添加或角色调整记录。

## Day 1 必测路径

每个试用用户都要完成：

- 登录、退出、重新登录。
- 修改昵称或头像。
- 向另一名试用用户发送好友申请，并完成同意。
- 发送一条单聊消息。
- 在试用群聊里发送一条消息。
- 上传一个小于 10 MB 的允许类型附件，并由另一名成员打开。
- 搜索联系人名、群名和中文消息关键词。
- 在个人资料页查看登录设备。
- Chrome/Edge 或 Android Chrome 用户开启 Web Push 通知，并确认能收到通用通知。

管理员额外完成：

- 添加成员。
- 调整 1 名成员的角色。
- 移除 1 名测试成员，并确认该成员刷新后不能继续读取群聊和附件。
- 确认 owner/admin 可见“群管理记录”，member 不可见。

## P0/P1 判定

P0：立即暂停扩容并优先修复。

- 生产站点不可用。
- 多数用户无法登录。
- 消息无法发送或大量丢失。
- 非成员能读取私聊、群聊、附件、push subscription 或群管理记录。
- 附件泄露或成员移除后仍可访问受限内容。

P1：记录复现步骤和负责人，先处理再扩容。

- 好友申请核心路径不可用。
- 群成员添加、移除或角色调整不可用。
- 附件上传或打开不可用。
- Web Push 完全不可用。
- 搜索、登录设备或群管理记录核心路径不可用。
- 移动端布局阻塞核心操作。

P2/P3：先收集，不阻塞先导。

- 文案不清楚。
- 体验不顺。
- 非核心浏览器兼容问题。
- 后续功能建议。

## 日报记录

每天复制 `docs/trial-daily-report-template.md`，填写：

- 试用人数和活跃群聊。
- 当日版本/tag。
- P0/P1 列表和处理状态。
- 核心路径状态。
- 群管理记录观察。
- 是否继续扩大试用。

## 扩容闸门

满足以下条件后，才从 5 人扩到 20-30 人：

- 先导 1-3 天无 P0。
- P1 均有复现步骤、处理决策和负责人。
- owner/admin 能独立添加、移除成员并理解权限影响。
- 非成员权限回归通过。
- 试用者已阅读安全说明。

扩容前必须再跑一次生产 smoke：登录、群成员管理、好友申请、单聊、群聊、附件、搜索、通知、登录设备、群管理记录。

## 修复流程

```text
复现 -> 最小修复 -> npm run lint -> npm run test -- --run -> npm run build -> push main -> Vercel Production smoke -> tag v0.6.x-beta
```

修复范围只包含稳定性、权限、通知、附件、成员管理和移动端阻塞问题，不夹带视频动态头像、频道、机器人或语音/视频功能。
