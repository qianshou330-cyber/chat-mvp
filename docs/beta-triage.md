# v0.2 Beta 反馈分诊指南

## 分诊节奏

外测期间建议每天查看一次新 Issue。

处理顺序：

1. 阻塞缺陷。
2. 安全或权限相关反馈。
3. 可复现的通知、附件、头像或消息收发问题。
4. 好友申请、单聊、群聊和搜索流程问题。
5. 体验反馈。
6. 功能建议。

## 标签

继续使用 GitHub 默认标签：

- `bug`：功能坏掉或验收路径失败。
- `question`：需要讨论的体验反馈。
- `enhancement`：功能建议或下一阶段候选项。
- `documentation`：测试说明、发布说明或总结文档。

如果没有专门标签，可以把严重程度写在 Issue 顶部：

```md
严重程度：P1
区域：单聊
状态：需要复现
```

## 严重程度

| 严重程度 | 含义 | 示例 |
| --- | --- | --- |
| P0 | 大多数测试者无法继续 | 无法注册、无法登录、生产页面空白 |
| P1 | 核心聊天路径被阻塞 | 消息失败、好友申请失败、附件无法打开、通知完全不可用 |
| P2 | 明显体验问题 | 文案看不懂、布局拥挤、空状态不清楚、搜索结果不清晰 |
| P3 | 后续优化 | iOS/Safari Push 兼容、v0.3 工作区、多端设备管理、视频动态头像 |

## 复现标准

一个缺陷进入修复前，至少需要记录：

- 测试者代号。
- 设备和浏览器。
- 使用的生产地址。
- 精确操作步骤。
- 期望结果。
- 实际结果。
- 涉及视觉问题时附截图或录屏。

如果是权限问题，还要记录该账号是否为会话成员。

如果是通知问题，还要记录浏览器通知权限状态、页面是否在后台、是否使用同一个账号多端登录。

## 修复流程

每个修复都按这个顺序：

1. 用测试账号在本地或生产环境复现。
2. 只修改最小必要范围。
3. 运行 `npm run lint`、`npm run test -- --run`、`npm run build`。
4. 推送到 `main`。
5. 等 GitHub/Vercel 状态通过。
6. Vercel 部署后做生产 smoke test。
7. 在对应 Issue 里评论验证结果。

## v0.3 候选项评分

至少收集 10 条反馈后再评分。

| 因素 | 分值 |
| --- | --- |
| 阻塞当前 beta 路径 | 3 |
| 2 名或更多测试者提到 | 2 |
| 能减少支持成本或困惑 | 2 |
| 能支撑 20-30 人公司试用 | 3 |
| 实现风险较高 | -2 |

默认候选顺序：

1. 团队/工作区。
2. 管理员成员管理。
3. 同账号多端登录管理。
4. 生产错误日志、反馈入口和测试数据清理流程。
5. 推送通知兼容性增强和服务端搜索优化。
